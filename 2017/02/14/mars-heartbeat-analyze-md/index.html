mars智能心跳过程解析

发表于 2017-02-14

做IM的都知道，要想保持这个底层通道连接存活，那就需要使用心跳策略，定时给服务器发送数据包来告诉服务器本地客户端还存活，防止服务器在一定时间后将这个socket连接给断开，但是如果频繁的发送数据包会造成用户电量大量消耗，导致我们的APP出现在耗电排行的前面，很容易被用户杀掉，那么如何控制这个发送时间间隔呢，对于这个问题，微信提出了他们的解决方案，智能心跳策略

基础概念

GCM:这个是Google Cloud Messageing 的缩写，即Google自己的消息推送通道，对于国外的APP来说，一般都会选择使用这个来进行消息的推送，但是在国内，因为Google play商店限制导致这个通道使用率很低，所以一般都不会使用

信令通道与数据通道: 下面是两段关于这个概念的介绍，
1, 手机要先建立一个通道，无线侧是RAB，核心侧是GTP隧道，这个建好后，微信建立个TCP连接，然后就发数据了；建立RAB和GTP的过程，就是信令控制过程
2, 手机有空闲模式和连接模式，连接模式下还有四种状态。每次手机发起联网申请，都会有模式的改变，模式的改变都是通过和移动网络间信令的传递来实现，把手机从空闲模式转换到连接模式，才能互相连通。就像拨打电话一样，从你拨打对方号码一直到对方的电话振铃，都是信令的消息在交互，而对方接听电话以后，那就是属于业务流量了。无论是你访问网页还是看视频还是用微信，从你发起联网申请一直到手机和服务器传递IP数据包以前，都是通过信令的交换来实现手机和移动网络间的连接
3, 智能手机之所以会造成信令风暴，最主要的原因就是根据其使用业务的不同情况，3GPP定义了几种不同的状态；智能手机在状态间迁移时，会产生不同数量的信令负荷；如果状态转换的频繁了，则将会产生大量信令，加重交换机的处理负担，超出处理能力时，会导致交换机宕机，也称为信令风暴。 微信恰恰是频繁引起状态转换的应用之一。 微信其实和其他IP之上的服务的区别在于比如观看视频，在手机从空闲状态进入CELL_DCH状态以后，就不会继续有信令发起，而微信呢，平均每个PS呼叫下载数据量非常低，绝大部分呼叫下载数据量低于5Kbyte，每次PS呼叫时间也很短，微信和QQ大量使用了即时消息、状态服务、位置服务等在线业务，终端均建立了PS RABs连接，但这些连接的数据传输量很小，如仅显示好友状态，不传输或者只传输少量的数据。智能手机为保持这些应用，频繁收发小数据包来保持IP连接开通，即“心跳”反应，通常间隔每分钟或几分钟，数据量远远小于1KB。另外为了延长电池使用时间，每次消息发送完毕后，终端强行转换为空闲状态，状态转换不受网络控制。终端节电功能造成的释放次数往往是系统正常释放次数的10倍以上。 手机有空闲模式和连接模式。从空闲状态转到cell_dch状态，空口大概需要三十条信令。连接模式也有四种状态，状态之间的转换都是需要消耗信令的。所以实际上所有的小流量包，频繁呼叫类型的应用都是引起信令风暴的原因，而为什么抓住微信呢，第一，微信用户多，第二，微信动了运营商语音和短信服务的奶酪。
